📝 Syntax & Grundkonstrukte

Hello World

Java

public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}

Kotlin

fun main() {
    println("Hello, World!")
}

Tinox

fn main() -> Int64
{
    println("Hello, World!");
    return 0;
}

TypeScript

console.log("Hello, World!");

Variablen & Inferenz

Java

final int x = 42;           // immutable
int y = 10;                  // mutable
var name = "Alice";          // Inferenz (Java 10+)
String msg = "Hi " + name + "!"; // kein Interpolation

Kotlin

val x: Int = 42              // immutable
var y: Int = 10              // mutable
val name = "Alice"           // Inferenz überall
val msg = "Hi $name!"        // String-Interpolation
val expr = "2+2=${2 + 2}"    // Ausdruck in ${}

Tinox

let x: Int64 = 42;           // immutable
var y: Int64 = 10;           // mutable
let name = "Alice";          // Inferenz überall
let msg = "Hi ${name}!";     // String-Interpolation

TypeScript

const x: number = 42;        // immutable
let y: number = 10;          // mutable
const name = "Alice";        // Inferenz
const msg = `Hi ${name}!`;   // Template Literal

Schleifen & Ranges

Java

for (int i = 0; i < 5; i++) {
    System.out.println(i);
}
for (String s : list) {
    System.out.println(s);
}
IntStream.range(0, 5)
         .forEach(System.out::println);

Kotlin

for (i in 0 until 5) println(i) // 0..4
for (i in 0..5) println(i)      // 0..5
for (s in list) println(s)

(0 until 5).forEach { println(it) }
(1..5 step 2).forEach { println(it) }

Tinox

for i in 0..5        // exklusiv: 0,1,2,3,4
{
    println(i);
}
for i in 0...5       // inklusiv: 0,1,2,3,4,5
{
    println(i);
}
for ch in "hello"    // Zeichen-Iteration
{
    print(ch);
}

TypeScript

for (let i = 0; i < 5; i++) {
    console.log(i);
}
for (const s of list) {
    console.log(s);
}
[...Array(5).keys()]
    .forEach(i => console.log(i));

Tinox — Syntax-Vorteile

  • Weniger Boilerplate als Java (kein public class Wrapper)
  • let/var macht Mutability sofort sichtbar
  • String-Interpolation direkt in der Sprache
  • Ranges (0..5, 0...5) — kein Iterator-API nötig
  • Allman-Style Formatter erzwingt einheitlichen Stil

Tinox — Syntax-Nachteile

  • Kotlin hat kürzere Lambdas und weniger Zeremonie
  • Kein REPL (noch in Planung)
  • Allman-Style kann polarisieren
  • Kleinere Community, weniger Tutorials

🧬 Typen & Generics

Generics

Java

// Type Erasure: T → Object zur Laufzeit
public class Box<T> {
    private T value;
    public Box(T value) { this.value = value; }
    public T get() { return value; }
}

Box<Integer> b = new Box<>(42);
// Wildcards: Box<? extends Number>
// Kein Box<int>, nur Box<Integer>

Kotlin

// Type Erasure + reified für Inline-Funktionen
class Box<T>(val value: T)

val b = Box(42)
println(b.value)

// Variance-Annotations:
// out T (covariant), in T (contravariant)
// reified bei inline fun:
inline fun <reified T> isType(x: Any) = x is T

Tinox

// Monomorphisierung: echter Typ bleibt erhalten
class Box<T>
{
    value: T;

    fn get() -> T
    {
        return this.value;
    }
}

let b = new Box<Int64>(42);
// Primitive direkt verwendbar (kein Boxing)

TypeScript

// Strukturelles Typsystem, Compile-Zeit gelöscht
class Box<T> {
    constructor(private value: T) {}
    get(): T { return this.value; }
}

const b = new Box<number>(42);
// Constraints: Box<T extends number>
// Conditional Types: T extends U ? X : Y

Collections

Java

List<String> list = new ArrayList<>();
list.add("hello");
list.size();
list.get(0);

Map<String, Integer> map = new HashMap<>();
map.put("a", 1);
map.get("a");

Kotlin

val list = mutableListOf("hello")
list.add("world")
list.size
list[0]

val map = mutableMapOf("a" to 1)
map["b"] = 2
map["a"]

// Read-only:
val ro = listOf(1, 2, 3)

Tinox

let arr = ["hello"];
arr.push("world");
arr.len();
arr[0];

let m = @{"a" => 1};
m.insert("b", 2);
m.get("a");
// typisiert:
let m: Map<String, Int64> = Map::new();

TypeScript

const list: string[] = [];
list.push("hello");
list.length;
list[0];

const map = new Map<string, number>();
map.set("a", 1);
map.get("a");

const obj: Record<string, number> = { a: 1 };

Tinox Generics — Vorteile

  • Monomorphisierung: kein Overhead durch Type Erasure
  • Primitive direkt als Typparameter (kein Integer vs. int)
  • Map-Literal-Syntax (@{"k" => v}) kompakter
  • Array-Builtins (first, last, join) ohne Import

Tinox Generics — Nachteile

  • Monomorphisierung kann Binärgröße erhöhen
  • Wildcards (? extends T) und Variance-Annotations fehlen
  • Kein reified / Reflection-Zugriff auf Typinformation
  • Bounded Type Parameters noch eingeschränkt

🏗️ OOP, Klassen & Vererbung

Vererbung

Java

public abstract class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }
    public abstract String speak();
}

public class Dog extends Animal {
    public Dog(String name) { super(name); }

    @Override
    public String speak() {
        return "Woof! I am " + name;
    }
}

Animal a = new Dog("Rex");
System.out.println(a.speak());

Kotlin

abstract class Animal(protected val name: String) {
    abstract fun speak(): String
}

class Dog(name: String) : Animal(name) {
    override fun speak() = "Woof! I am $name"
}

val a: Animal = Dog("Rex")
println(a.speak())

Tinox

class Animal
{
    name: String;

    fn speak() -> String
    {
        return "...";
    }
}

class Dog extends Animal
{
    fn speak() -> String
    {
        return "Woof! I am ${this.name}";
    }
}

let a: Animal = new Dog();
a.name = "Rex";
println(a.speak());

TypeScript

abstract class Animal {
    constructor(protected name: string) {}
    abstract speak(): string;
}

class Dog extends Animal {
    speak(): string {
        return `Woof! I am ${this.name}`;
    }
}

const a: Animal = new Dog("Rex");
console.log(a.speak());

Data Classes & Records

Java

// Java 16+: Records
public record Point(int x, int y) {}

// auto-generiert: equals, hashCode,
// toString, Accessor-Methoden x(), y()

Point p = new Point(3, 4);
System.out.println(p.x()); // 3

Kotlin

// data class: equals, hashCode,
// toString, copy() auto-generiert
data class Point(val x: Int, val y: Int)

val p = Point(3, 4)
println(p.x) // 3

// Kopieren mit Änderung:
val p2 = p.copy(y = 10)

Tinox

// unmodifiable: immutable Wertobjekt
// new() wird auto-generiert
unmodifiable Point(x: Int64, y: Int64)

let p = Point.new(3, 4);
println(p.x); // 3

// Felder via FieldAccess (read-only)

TypeScript

// Kein eingebautes Record-Konzept
// Interface + Object als Workaround:
interface Point { readonly x: number; readonly y: number; }

const p: Point = { x: 3, y: 4 };
console.log(p.x); // 3

// Klassen-Alternative mit readonly:
class Point {
    constructor(readonly x: number, readonly y: number) {}
}

Interfaces

Java

public interface Printable {
    String toString();
    // Default-Methoden möglich:
    default void print() {
        System.out.println(toString());
    }
}

public class Point implements Printable {
    int x, y;
    @Override
    public String toString() {
        return "(" + x + ", " + y + ")";
    }
}

Kotlin

interface Printable {
    fun label(): String
    // Default-Implementierung möglich:
    fun print() = println(label())
}

data class Point(val x: Int, val y: Int) : Printable {
    override fun label() = "($x, $y)"
}

// Extension Function (ohne Interface):
fun Point.describe() = "x=$x y=$y"

Tinox

interface Printable
{
    fn toString() -> String;
}

class Point implements Printable
{
    var x: Int64;
    var y: Int64;

    fnc new(x: Int64, y: Int64) -> Point
    {
        return Point { x: x, y: y };
    }

    fn toString() -> String
    {
        return "(${this.x}, ${this.y})";
    }
}

TypeScript

interface Printable {
    toString(): string;
}

class Point implements Printable {
    constructor(
        public x: number,
        public y: number
    ) {}

    toString(): string {
        return `(${this.x}, ${this.y})`;
    }
}
// Strukturelles Typing: Duck Typing

Kotlin OOP — Vorteile

  • data class generiert equals/hashCode/copy automatisch
  • Extension Functions ohne Vererbung oder Wrapper
  • Sealed Classes für exhaustives Pattern Matching
  • Kein Boilerplate für Konstruktoren (primary constructor)
  • Volle Java-Interoperabilität

Tinox OOP — Vorteile

  • fn (Instanz) vs. fnc (statisch) sofort sichtbar
  • unmodifiable für immutable Wertobjekte
  • Vtable-basierter Interface-Dispatch
  • Weniger Boilerplate als Java

λ Funktionales Programmieren

Lambdas & Higher-Order-Funktionen

Java

// Funktionstypen via java.util.function.*
Function<Integer, Integer> doubled = n -> n * 2;

static int apply(int x,
    Function<Integer, Integer> f) {
    return f.apply(x);
}
apply(21, n -> n * 2); // 42

List.of(1,2,3).stream()
    .map(n -> n * 2)
    .filter(n -> n > 2)
    .toList();

Kotlin

// Funktionstypen nativ: (Int) -> Int
val doubled: (Int) -> Int = { n -> n * 2 }
// oder Kurzform mit it:
val doubled2: (Int) -> Int = { it * 2 }

fun apply(x: Int, f: (Int) -> Int) = f(x)
apply(21) { it * 2 } // 42 (trailing lambda)

listOf(1, 2, 3)
    .map { it * 2 }
    .filter { it > 2 }

Tinox

fn apply(x: Int64,
    f: fnc(Int64) -> Int64) -> Int64
{
    return f(x);
}

let result = apply(21, n => n * 2); // 42

// Lambda als Wert
let square: fnc(Int64) -> Int64
    = fn(x: Int64) -> Int64
{
    return x * x;
};

TypeScript

const doubled = (n: number) => n * 2;

function apply(x: number,
    f: (n: number) => number): number {
    return f(x);
}
apply(21, n => n * 2); // 42

[1, 2, 3]
    .map(n => n * 2)
    .filter(n => n > 2);

Pattern Matching

Java

// Java 21: Pattern Matching switch
Object obj = ...;
String result = switch (obj) {
    case Integer i -> "int: " + i;
    case String s  -> "str: " + s;
    default        -> "other";
};

// Enum switch
Direction d = Direction.NORTH;
switch (d) {
    case NORTH -> System.out.println("N");
    case SOUTH -> System.out.println("S");
    default    -> {} // nötig
}

Kotlin

// when als Ausdruck (exhaustiv bei sealed)
sealed class Shape
class Circle(val r: Double) : Shape()
class Rect(val w: Double, val h: Double) : Shape()

fun area(s: Shape) = when (s) {
    is Circle -> Math.PI * s.r * s.r
    is Rect   -> s.w * s.h
    // kein else nötig — sealed ist exhaustiv
}

// when ohne Argument als if-Kette:
when {
    x < 0  -> println("negativ")
    x == 0 -> println("null")
    else   -> println("positiv")
}

Tinox

enum Direction
{
    North; South; East; West;
}

fnc turn(d: Direction) -> Direction
{
    match d
    {
        North => return East;
        East  => return South;
        South => return West;
        West  => return North;
    }
}
// Exhaustiveness wird geprüft

TypeScript

type Direction = "North"|"South"|"East"|"West";

function turn(d: Direction): Direction {
    switch (d) {
        case "North": return "East";
        case "East":  return "South";
        case "South": return "West";
        case "West":  return "North";
    }
}

// Discriminated Unions:
type Shape =
    | { kind: "circle"; r: number }
    | { kind: "rect"; w: number; h: number };

Kotlin Funktional — Vorteile

  • Trailing-Lambda-Syntax: f(x) { it * 2 } sehr lesbar
  • scope functions: let, run, with, apply, also
  • Sealed Classes + when = vollständiges Pattern Matching
  • Sequence für lazy evaluation (kein Stream nötig)

Tinox Funktional — Vorteile

  • Funktionstypen direkt (kein java.util.function Import)
  • match ist exhaustiv bei Enums
  • Kompakte Lambda-Syntax (n => n * 2)
  • defer für sauberes Resource-Management

⚡ Concurrency & Async

Java

// Virtual Threads (Java 21)
try (var exec =
    Executors.newVirtualThreadPerTaskExecutor()) {
    exec.submit(() -> process());
}

// CompletableFuture
CompletableFuture<Integer> f =
    CompletableFuture
        .supplyAsync(() -> fetchData(21))
        .thenApply(n -> n * 2);
int result = f.get(); // blockiert

// Channel: BlockingQueue
BlockingQueue<Integer> ch =
    new LinkedBlockingQueue<>();
ch.put(99);
int v = ch.take();

Kotlin

// Coroutines: leichtgewichtige Threads
suspend fun fetchData(id: Int): Int {
    delay(100) // non-blocking
    return id * 2
}

fun main() = runBlocking {
    val result = async { fetchData(21) }
    println(result.await()) // 42

    // Channels (Go-ähnlich)
    val ch = Channel<Int>()
    launch { ch.send(99) }
    println(ch.receive()) // 99

    // Flow für reaktive Streams
    flow { emit(1); emit(2) }
        .collect { println(it) }
}

Tinox

async fn fetchData(id: Int64) -> Int64
{
    return id * 2;
}

fn main() -> Int64
{
    let handle = spawn fetchData(21);
    let result = await handle; // 42
    println(result);

    // Channels (Go-inspiriert)
    let ch = channel;
    send ch -> 99;
    let v = recv ch;
    println(v);

    return 0;
}

TypeScript

async function fetchData(id: number)
    : Promise<number> {
    return id * 2;
}

async function main() {
    const result = await fetchData(21); // 42
    console.log(result);

    // Parallele Ausführung
    const [a, b] = await Promise.all([
        fetchData(1),
        fetchData(2),
    ]);

    // Kein nativer Channel
}

Kotlin Concurrency — Vorteile

  • Coroutines: leichtgewichtig, kein Thread-Overhead
  • structured concurrency (CoroutineScope)
  • Flow für reaktive Datenströme
  • Channels und Select nativ verfügbar

Tinox Concurrency — Vorteile

  • async/await direkt in der Sprache (kein Framework nötig)
  • Channels als First-Class-Konzept
  • spawn ist intuitiver als CompletableFuture-Ketten
  • Kein Import aus java.util.concurrent

🚀 Memory Management & Performance

Java

// JVM-basiert: alles auf dem Heap
// GC: G1, ZGC, Shenandoah
// Startup: ~100–500ms (JVM-Start)
// Peak-Performance: sehr hoch (JIT)
// Memory: JVM-Heap + MetaSpace
// GraalVM native-image als Option

Kotlin

// JVM: identisch zu Java (gleicher GC)
// Kotlin/Native: LLVM-Backend, kein GC (ARC)
// Kotlin/JS: transpiliert zu JavaScript

// JVM-Startup: ~150–500ms
// Native-Startup: instant
// Native: kein JVM-Overhead
// Native GC: ARC (Automatic Reference Counting)

Tinox

// LLVM-Backend: AOT zu Maschinencode
// GC: eigene Tinox Runtime (C, pthread)
// Startup: instant (native Binary)
// Performance: vergleichbar mit C
// Memory: kein JVM-Overhead
// Kein GraalVM nötig

TypeScript

// Transpiliert zu JS → V8 (Node.js)
// GC: V8 generational GC
// Startup: ~50–200ms (Node.js)
//          sehr schnell mit Bun
// Performance: gut für I/O-Workloads
// Memory: V8-Heap-Overhead

Tinox — Performance-Vorteile

  • Kein JVM-Start-Overhead (instant startup)
  • LLVM O2/O3 direkt anwendbar
  • Kein Boxing/Unboxing bei Primitiven in Generics
  • Kein MetaSpace-Overhead
  • Ideal für CLI-Tools, Microservices, Embedded

Tinox — Performance-Nachteile

  • JVM-JIT nach Warmup oft sehr hoch (Java/Kotlin)
  • Kotlin/Native bietet ebenfalls instant startup
  • Tinox GC noch nicht so ausgereift wie G1/ZGC
  • Kein Profiling-Tooling wie Java Flight Recorder

Tinox Stdlib — Vorteile

  • JSON, HTTP-Server, JWT, bcrypt ohne externe Dependencies

  • Cron, PubSub, Cache out-of-the-box

Tinox Stdlib — Nachteile

  • Kein Datenbankzugriff (kein JDBC/ORM-Äquivalent)
  • Kein Spring/Ktor-Ökosystem
  • Kein zentrales Repository für Third-Party-Libraries
  • Kein Multi-Platform-Target wie Kotlin

Tinox Paket-Manager — Vorteile

  • YAML statt XML — deutlich lesbarer und kompakter
  • URL-basiert: kein zentrales Repository nötig (self-hosted)
  • Alles im selben CLI (kein separates mvn/gradle)
  • tinox package erzeugt verteilbares tar.gz direkt
  • Deps projekt-lokal (.tinox/deps/) — kein globaler Cache-Konflikt

Tinox Paket-Manager — Nachteile

  • Keine Checksum-/Signaturverifikation (noch)
  • Keine Version-Conflict-Resolution

🔒 Fehlerbehandlung & Null Safety

Java

// Checked vs. Unchecked Exceptions
int divide(int a, int b)
    throws ArithmeticException {
    if (b == 0)
        throw new ArithmeticException("Div/0");
    return a / b;
}

try {
    int r = divide(10, 0);
} catch (ArithmeticException e) {
    System.err.println(e.getMessage());
} finally { cleanup(); }

// Null Safety: fehlt
Optional<String> opt = Optional.ofNullable(s);
opt.ifPresent(System.out::println);

Kotlin

// Keine Checked Exceptions
fun divide(a: Int, b: Int): Int {
    if (b == 0) throw ArithmeticException("Div/0")
    return a / b
}

try {
    val r = divide(10, 0)
} catch (e: ArithmeticException) {
    println(e.message)
} finally { cleanup() }

// Null Safety im Typsystem:
var s: String? = null
println(s?.length)   // safe call → null
println(s ?: "leer") // Elvis-Operator
s!!.length           // force unwrap (NPE möglich)

Kotlin Fehlerbehandlung — Vorteile

  • Null Safety direkt im Typsystem (T?)
  • Elvis-Operator (?:) für kompakte Fallbacks
  • Keine Checked Exceptions (kein throws-Chaos)
  • runCatching { } als funktionaler Wrapper

Tinox Fehlerbehandlung — Vorteile

  • defer ist eleganter als try-with-resources
  • Kein Checked-Exception-Chaos
  • Option<T> / Result<T> in Stdlib verfügbar
  • LIFO-Reihenfolge mehrerer defers ist intuitiv

🏆 Fazit & Empfehlung

Wann ist was die bessere Wahl?

Java verwenden, wenn…

du ein großes Team, ein riesiges Ökosystem (Spring, Hibernate, JEE) oder Unternehmens-Software mit langfristigem Support brauchst. Datenbankintegration, Microservices-Orchestrierung (Kubernetes, Spring Cloud) und breites Tooling (IntelliJ, APM) sind entscheidend.

Kotlin verwenden, wenn…

du die JVM-Plattform willst, aber moderne Sprachfeatures schätzt: Null Safety, Coroutines, data classes, Extension Functions. Ideal für Android-Entwicklung, Server-Side mit Ktor, und Teams die von Java migrieren. Volle Java-Interop inklusive.

Tinox verwenden, wenn…

du native Performance ohne JVM-Overhead willst, kleine bis mittlere Services oder CLI-Tools baust, oder die Java-ähnliche Syntax mit weniger Boilerplate schätzt. Ideal für instant-startup-Szenarien, Eigenentwicklungen und Lernprojekte.

TypeScript verwenden, wenn…

du Web-Apps, APIs oder Full-Stack-Projekte mit dem riesigen npm-Ökosystem baust. Frontend und Backend teilen dieselbe Sprache, IDE-Support ist exzellent, und das Team hat JavaScript-Background.

Tinox ist eine moderne Sprache in aktiver Entwicklung — Paket-Manager, LSP, REST-Framework und 50+ Stdlib-Module sind verfügbar. Weitere Features (Debugger, REPL, zentrales Repository) werden in zukünftigen Versionen ergänzt.
